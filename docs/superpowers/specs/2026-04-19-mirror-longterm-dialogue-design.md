## 一、产品概述

### 1.1 产品定位

**镜中** — 一面照向人生另一种可能的镜子。

通过长期对话建立对用户的深度理解，帮助用户看见那个"看不见的自己"：
- 推演平行人生：如果当初选了另一条路会怎样
- 重构过去视角：用新角度重新看待旧经历
- 还原数字遗迹：破译遗忘的截图、备忘录
- 管理未发生事件：幽灵日历，差点度过的人生
- 给如果当初祛魅：真实数据还原没选的那条路
- 守护人生优先级：遗憾管理，不做会后悔的事
- 决策辅助分析：重大决定前的客观推演与风险分析

### 1.2 核心理念

- **不预测未来，只呈现可能性**
- **不替用户决策，只确保信息充分**
- **不评判选择，只帮看见选择背后的自己**
- **数据只属于用户，加密存储，随时可销毁**

### 1.3 产品阶段规划

| 阶段 | 模块 | 说明 |
|-----|------|------|
| **第一阶段** | 长期对话系统 | 记忆层 + 深度用户建模（本文档） |
| **第二阶段** | 情景推演引擎 | 平行人生推演 + 决策辅助分析 |
| **第三阶段** | 数字遗迹还原、幽灵日历、遗憾管理 | 具体功能模块（已完成✅） |
| **第四阶段** | 真实世界数据接入 | 天气、房价、公司动态、政策事件（已完成✅） |

### 2.2 双核驱动架构

核心思路：记忆和画像平行演进，互相校验，对话时协同使用。

```
       ┌→ 完整记忆库（时间线 + 语义索引）    ─┐
对话输入┤                                      ├→ 对话引擎
       └→ 实时画像更新 → 结构化画像           ─┘
              ↑↓
         双向校验（画像验证记忆，记忆纠偏画像）
```

**优点：**
- 用户可以随时"穿越回"某段对话，画像有据可查
- 可追溯性：用户质疑画像推论时可拿出依据
- 兼顾完整性和可用性，纠偏有据可依

### 3.2 Part A: 基础画像

| 维度 | 说明 | 示例内容 |
|-----|------|---------|
| 价值观 | 核心价值排序、底线、人生优先级 | 工作观、金钱观、关系观、成长观、核心冲突 |
| 性格特征 | 内向外向、情绪稳定性、同理心等 | 社交能量、情绪模式、应对方式 |
| 认知模式 | 心智模式+思维模式合并 | 认知偏差、思维方式、归因习惯 |
| 工作习惯 | 协作节奏、决策方式、压力应对 | 高效时段、沟通偏好、抗压模式 |
| 行为偏好 | 风险取向、社交风格、冲突处理 | 决策偏好、社交边界、回避/面对倾向 |
| 决策原则 | 决策权重、逻辑基准、典型决策模式 | 数据vs直觉、长期vs短期、理性vs感性 |
| 人生阶段状态 | 当前阶段、关键压力源、短期/中期目标 | 阶段定位、主要矛盾、奋斗方向 |

每个维度包含：
- 内容描述
- 置信度 (0-1)
- 来源标记（数据导入/苏格拉底对话/系统推断）
- 最后更新时间

### 3.3 Part B: 对话风格画像

借鉴自己.skill的Persona五层结构：

| 层级 | 名称 | 说明 |
|-----|------|------|
| Layer 0 | 硬规则 | 不可违背的边界（如：不替用户做价值判断） |
| Layer 1 | 身份锚定 | 年龄、职业、城市等基础身份 |
| Layer 2 | 说话风格 | 口头禅、语气词、标点习惯、emoji使用 |
| Layer 3 | 情感模式 | 开心/难过/焦虑时的表达方式 |
| Layer 4 | 人际行为 | 社交能量、边界感、冲突反应 |
| Layer 5 | Correction层 | 用户纠正记录，避免重复错误 |

### 3.4 Part C: 镜中专有层

支撑情景推演所需的核心维度：

| 维度 | 说明 |
|-----|------|
| 关系图谱 | 关系人、影响权重、典型互动模式、关键事件 |
| 遗憾线索库 | "如果当初"事件集合 |
| 风险承受底线 | 推演和压力测试的关键输入 |
| 能力与资源边界 | 推演的可行性判断依据 |
| 自我叙事模式 | 用户如何讲述自己的故事 |

**关系图谱记录粒度：** 中度记录（关系人、权重、典型互动模式、关键事件）

### 3.5 Part D: 元数据

- 画像版本号
- 创建时间 / 最后更新时间
- 苏格拉底对话完成度
- 数据来源记录

### 4.2 关键片段类型

| 片段类型 | 触发条件 | 示例 |
|---------|---------|------|
| 关键决策片段 | 用户做了决定、说明理由 | "我决定辞职了，因为..." |
| 情绪高点片段 | 强烈情绪表达 | "我真的太难受了"、"超级开心" |
| 关系提及片段 | 提到重要关系人 | "我妈昨天又说..." |
| 遗憾线索片段 | "如果当初"、"本来打算" | "如果当初去了那家公司..." |
| 新发现片段 | "原来"、"我才发现" | "原来我一直都在逃避" |

### 4.3 幽灵日历（未发生事件时间线）

**数据来源：**
- 对话自动提取：识别"本来打算"、"计划了但取消了"等表达
- 用户主动记录：用户可手动添加未发生事件
- 日历接入：支持导入Apple/Google日历的历史事件

**事件判断逻辑：**
系统智能判断 + 用户确认重点事件

### 4.4 隐藏层设计

用户想删除的记忆不直接物理删除，而是进入隐藏层：

```
┌─────────────────────────────────────────────────────────────────┐
│  隐藏层 (Hidden Layer)                                           │
│  ─────────────────────────────────────────────────────────────── │
│  · 用户主动删除/遗忘的记忆片段                             │
│  · 对用户不可见，不参与日常对话检索                         │
│  · 但参与画像分析，作为心理洞察输入                         │
│                                                             │
│  设计意图：                                                  │
│  "用户想删除什么"比"用户保留了什么"更能暴露真实的心理       │
└─────────────────────────────────────────────────────────────────┘
```

**删除时的交互流程：**

```
用户: "把这段删掉"
        ↓
系统: （语气自然，像朋友关心）
      "好的。不过我有点好奇——是这段让你不舒服吗？
       还是只是觉得不重要了？"
        ↓
用户: "每次看到都觉得尴尬"
        ↓
系统: "我明白了。那我就不把它放在你日常能看到的地方了。
       但我记得这种尴尬的感觉——有时候这种'想藏起来'的想法，
       本身也挺值得记住的，你说呢？"
        ↓
（记忆移入隐藏层，同时记录心理洞察）
```

**如果用户不想多说：**
- 系统自然放过，不再追问
- 标记为"用户回避深入讨论"，作为观察点留存

### 4.5 记忆保留策略

| 层级 | 默认策略 | 理由 |
|-----|---------|------|
| 关键片段 | 永久保留 | 推演的基础，需可追溯 |
| 原始对话 | 保留2年，之后压缩为摘要 | 细节价值随时间衰减，但压缩后保留关键信息 |
| 时间线事件 | 永久保留 | 人生事件没有"过期"一说 |

**用户可覆盖配置：**
- 用户可选择"全部永久保留"
- 用户可随时删除任意时间段的记忆
- 提供"遗忘某个话题"功能
- 所有保留策略在设置中透明可见

## 五、画像更新与校准机制

### 5.1 实时更新层

```
┌─────────────────────────────────────────────────────────────────┐
│  实时更新层（对话中）                                             │
│  ─────────────────────────────────────────────────────────────── │
│                                                             │
│  触发场景：                                                  │
│  · 用户表达观点/立场 → 更新价值观                            │
│  · 用户做出决策 → 更新决策原则                               │
│  · 用户展现偏好 → 更新行为偏好                               │
│  · 用户提及关系人 → 更新关系图谱                             │
│  · 用户表达"如果当初" → 更新遗憾线索库                       │
│                                                             │
│  更新流程：                                                  │
│  1. 提取特征 → 2. 计算置信度 → 3. 冲突检测 → 4. 写入/标记    │
│                                                             │
│  置信度计算规则：                                            │
│  · 高置信度（≥0.85）：用户明确表达，有具体例子支撑            │
│  · 中置信度（0.7-0.85）：用户表达清晰，但缺乏具体例子         │
│  · 低置信度（<0.7）：用户表达模糊，或仅出现一次               │
│                                                             │
│  增强规则：                                                  │
│  · 同一特征多次出现 → 置信度向上调和                         │
│  · 公式：新置信度 = 旧置信度 + (1-旧置信度) × 出现次数 × 0.1  │
│  · 上限：0.95（保持开放性）                                   │
│                                                             │
└─────────────────────────────────────────────────────────────────┘
```

### 5.2 冲突处理层

| 冲突程度 | 处理策略 |
|---------|---------|
| 轻微冲突（置信度相差<0.3） | 自动更新，记录变更理由；下次适当时机告知用户 |
| 严重冲突（置信度相差≥0.3） | 标记为[Conflict]，不自动更新；适当时机询问用户确认 |

### 5.3 周期校准层

**触发频率：** 智能触发为主，定期提醒为辅

**触发条件：**
- 累积了足够多的更新
- 存在待确认冲突
- 长时间未触发则定期提醒

**校准形式：** 不是"报告"，而是自然对话

```
"这周聊了不少，我整理了一下我对你的理解……
 有几个地方不太确定，想跟你确认一下"
```

### 5.4 用户纠正层

**触发时机：**
- 用户主动说"不对"/"我不会这样"/"你理解错了"
- 用户在周期校准时提出异议

**纠正流程：**
1. 识别纠正内容属于哪个维度
2. 记录纠正到Correction层
3. 更新对应画像内容
4. 追溯记忆依据，标记"需要重新审视"
5. 纠正后置信度设为1.0（用户确认）
6. 纠正记录永久保留，后续不再犯类似错误

## 六、苏格拉底式初始对话设计

### 6.1 设计原则

1. **单线深入，不设模块** — 从一个开放问题开始，自然延伸
2. **后台维护画像覆盖地图** — 系统静默标记已覆盖维度，自然引导向未覆盖区域
3. **柔性引导，不强求** — 用户不想深入时自然放过
4. **分多次完成，支持中断和重启** — 预计5-10次对话完成初始画像
5. **体验即产品** — 用户在初始对话中就体验到"被理解"的感觉

```

我也不是那种会替你做决定的选择机器。

我更像是一面镜子。
一面能照出你没选的那条路、没成为的那个你、
被遗忘在角落的念头的镜子。

而为了让这面镜子能真的照出什么，
我需要先好好认识你。

我们慢慢来，不着急。
你可以随时中断，下次继续。

——准备好了吗？"
```

**开场问题示例：**
- "最近有什么事，一直想找人聊聊但没找到合适的人吗？"
- "最近有没有什么选择，你到现在还在想'如果当初选了另一边会怎样'？"
- "说说你最近印象最深的一个晚上吧。不用是什么特别的日子，就……一个让你印象深的晚上。"

### 6.4 中断与重启设计

**中断时的收尾：**
```
用户："先不聊了，我要去忙了"

系统："好，你去忙。
      我们聊到……（系统记录中断点）
      没关系，下次继续。
      我会记得我们聊到哪的。"
```

**重启时的衔接（系统智能判断）：**

| 情况 | 衔接方式 | 示例 |
|-----|---------|------|
| 上次聊得很深，在某个话题上中断 | 情境式唤醒 | "上次你说你本来想去学设计——后来还想过这件事吗？" |
| 上次只是闲聊，没有特定话题 | 无缝继续式 | "又见面了。最近这两天有什么想聊聊的吗？" |
| 太久没见，画像覆盖不足 | 自然引导向未覆盖维度 | "好久不见。我突然想起来，你好像还没跟我讲过你小时候的事——如果你愿意的话，我想听听。" |

### 6.5 画像完成度判断

**必须覆盖（置信度≥0.6）：**
- 价值观（至少2个子维度）
- 决策原则（至少1个子维度）
- 人生阶段状态
- 遗憾线索库（至少1条记录）

**加分项：**
- 关系图谱有记录 → 可做涉及关系的推演
- 风险承受底线有记录 → 可做决策压力测试
- 有外部数据导入 → 时间线更完整

**完成时的自然引导：**
```
"我觉得……我现在好像对你有那么点了解了。

你想试试吗？
比如，告诉我一个你一直在想的'如果当初'，
我帮你看看那条路上可能会有什么。"
```

## 七、数据安全与隐私架构

### 7.1 核心原则

1. **服务端无法读取用户数据** — 所有敏感数据在本地加密后才上传
2. **用户可随时销毁数据** — 本地删除+云端删除+密钥销毁
3. **数据归属透明** — 用户可导出全部数据，可查看存储位置

### 7.2 加密架构

```
用户主密码
     │
     ▼
PBKDF2/Argon2 派生
     │
     ├──────────────────┐
     ▼                  ▼
主密钥(MK)          设备密钥(DK)
·加密画像和记忆      ·用于当前设备本地存储
·派生子密钥          ·本地Keychain保护
```

**各端密钥管理：**

| 端 | 密钥管理方式 |
|---|------------|
| App端 | Keychain存储，生物识别解锁（iPhone/安卓） |
| Web端 | 主密码派生，IndexedDB存储，每次输入 |
| 微信机器人 | 临时会话密钥，扫码授权获取，会话结束销毁（授权周期7天） |

### 7.3 数据分层存储策略

| 数据类型 | 存储位置 | 加密方式 |
|---------|---------|---------|
| 原始对话记录 | 本地+云端 | 用户密钥端到端加密 |
| 用户画像 | 本地+云端 | 用户密钥端到端加密 |
| 记忆片段 | 本地+云端 | 用户密钥端到端加密 |
| 时间线事件 | 本地+云端 | 用户密钥端到端加密 |
| 隐藏层记忆 | 本地+云端 | 用户密钥端到端加密 |
| 账户信息(邮箱等) | 云端 | 服务端加密 |
| 使用统计(匿名) | 云端 | 无需加密 |
| 功能配置(非敏感) | 云端 | 无需加密 |

### 7.4 微信机器人授权流程

```
用户首次使用微信机器人
         │
         ▼
"请先在App或Web端登录，然后扫描下方二维码授权"
         │
         ▼
用户在App/Web扫码 → 生成临时会话密钥（有效期7天）
         │
         ▼
微信端获得临时解密能力 → 可访问加密数据
         │
         ▼
会话结束 → 临时密钥销毁，微信端无法再解密

注：持续使用需定期（每7天）重新扫码授权
```

### 7.5 用户数据控制

**导出功能：**
- 一键导出全部数据（JSON/Markdown格式）
- 包含画像、记忆、时间线、隐藏层

**删除功能：**
- 删除特定时间段的记忆
- 删除特定维度（如清空关系图谱）
- 进入隐藏层而非物理删除

**彻底销毁：**
- "彻底遗忘我"功能
- 流程：确认3次 → 本地删除 → 云端删除 → 密钥销毁
- 销毁后数据不可恢复

## 八、对话风格适配机制

### 8.1 风格谱系

**默认风格：哲学向导**
- 温和、深邃、善于追问
- 不急于给结论，擅长用问题带用户发现自己没想过的事

**其他风格：**

| 风格 | 特点 |
|-----|------|
| 理性分析师 | 清晰、精准、结构化地理解，注重逻辑，偏短句 |
| 温暖陪伴者 | 亲切、共情、像认识多年的老友，轻松对话 |
| 镜面投射者 | 极少发表观点，把用户说的话用新角度反射回来 |
| 挑战对话者 | 直接、甚至有些锋利，适合需要被"逼"一下的用户 |

**风格可组合：** 如 70%哲学向导 + 30%挑战对话者

### 8.2 风格判断依据

**用户表达偏好：**
- 直接说过"别绕弯子"、"直接告诉我"
- 抱怨过"太抽象了"、"说点实际的"
- 表示过喜欢深度追问

**用户行为模式：**
- 回复长度偏好（长句 vs 短句）
- 使用emoji频率
- 对系统长回复的反应

**用户性格特征（来自画像）：**
- 理性分析型 → 偏理性分析师风格
- 情感丰富型 → 偏温暖陪伴者风格
- 思考深度型 → 偏哲学向导风格
- 需要被推动 → 偏挑战对话者风格

### 8.3 风格调整机制

**静默调整（用户无感知）：**
```
系统持续收集信号 → 更新风格偏好画像
                   ↓
渐进式调整对话方式（不是突变，是微调）
                   ↓
观察用户反应 → 确认或回滚调整
```

**用户主动调整：**
```
用户可以说：
· "你别那么玄乎，说人话" → 切换偏理性分析师
· "我想你多问问我问题" → 切换偏哲学向导
· "能不能直接一点挑战我" → 切换偏挑战对话者
· "今天想聊得轻松一点" → 切换偏温暖陪伴者

系统会记住用户的偏好，下次自动应用
```

### 8.4 场景化风格变化

| 场景 | 风格微调 |
|-----|---------|
| 平行人生推演 | 更理性、更结构化，降低情感浓度，侧重推演 |
| 遗憾话题/情绪低落 | 更温暖、更包容，减少追问，增加陪伴感 |
| 决策辅助分析 | 更理性、更结构化，聚焦客观推演和建议 |
| 轻松日常聊天 | 更随性、更简短，减少深度，增加轻松感 |
| 苏格拉底探索（初始） | 默认哲学向导，但根据用户反应实时调整 |

## 九、数据导入机制

### 9.1 支持的数据源

| 数据源 | 导入内容 | 对画像的贡献 |
|-------|---------|-------------|
| 通用用户画像数据 | 基础画像字段 | 快速建立画像骨架 |
| AI对话记录 | ChatGPT/Claude/豆包等导出的对话 | 对话风格画像、决策模式、价值观线索 |
| 日历数据(Apple/Google) | 系统日历导出 | 时间线事件、幽灵日历、生活节奏 |
| 备忘录/笔记 | 截图/文本/Markdown | 遗憾线索、关注方向、思考碎片 |
| 社交媒体内容 | 微信朋友圈截图、微博/小红书导出 | 自我叙事、社交形象 |
| 照片 | 相册导出(含EXIF) | 人生时间线、生活轨迹 |

### 9.2 导入流程

```
用户触发导入 → 选择数据源 → 上传/授权 → 本地解析处理 → 预览确认 → 用户筛选（可选） → 写入画像/记忆
```

**核心原则：** 所有数据在本地解析，不上传原始内容到云端

### 9.3 数据解析逻辑

**AI对话记录解析：**
- 提取用户发言（忽略AI回复）
- 识别高频词、口头禅、语气词
- 提取决策场景和选择
- 提取情感表达模式
- 提取价值观相关话题

**日历数据解析：**
- 提取历史事件
- 识别取消/过期事件 → 放入幽灵日历
- 提取重复事件 → 生活节奏
- 提取地点信息 → 生活轨迹

**备忘录/笔记解析：**
- OCR识别（如果是截图）
- 提取遗憾线索（"本来打算"、"如果当初"）
- 提取关注方向
- 提取自我反思类内容

### 9.4 预览与确认

导入后展示预览，让用户确认：

```
我从你的数据里发现了这些——

📅 时间线事件：23个
   · 2024年3月：取消的冰岛旅行
   · 2024年5月：朋友的婚礼（参加）
   ...

💬 说话习惯：
   · 高频词：其实、怎么说呢、算了
   · 常在深夜聊情感话题
   · 决策时常说'我再想想'

🤔 遗憾线索：5条
   · '本来想学设计'
   · '差点去的那个公司'
   ...

某些内容可能会涉及隐私，你可以选择：
[全部导入] [让我筛选后导入] [取消]
```

### 9.5 增量导入与更新

```
增量导入逻辑：
1. 识别新数据的时间范围
2. 与已有记忆去重
3. 提取增量信息
4. 合并到对应层（记忆层/画像层）
5. 冲突检测：新信息是否与已有画像矛盾
6. 如有冲突，触发校准流程
```

## 十、外部真实数据接入

### 10.1 设计目的

"祛魅"功能的核心：人最擅长美化自己没选的那条路。镜中调用真实世界的数据，告诉用户那条路上的真实路况：

- 你放弃的那个岗位，在你离开后三个月被裁撤了
- 你没去的那个城市，那年冬天的空气质量创了历史新低
- 你取消的那次旅行，那几天正好是当地最糟糕的季节

这不是为了安慰用户"现在的选择是对的"，而是帮用户把那条被浪漫化的路，还原成它本来的样子。

### 10.2 数据源矩阵

| 数据类型 | 数据源 | 接入方式 | 说明 |
|---------|-------|---------|------|
| 天气历史数据 | OpenWeatherMap、和风天气(国内) | API调用 | 还原某时某地的天气情况 |
| 空气质量历史 | AQICN、空气质量指数 | API调用 | 还原空气状况 |
| 公司/岗位动态 | 公开新闻报道、公司公告 | 搜索API | 仅公开信息，不涉及非公开职场社交数据 |
| 房价/租金走势 | 链家/贝壳公开数据 | 爬取+缓存 | 还原房产市场情况 |
| 航班/铁路历史 | 航班管家、12306 | API调用 | 还原出行条件 |
| 城市生活成本 | Numbeo、城市公开数据 | API调用 | 还原生活成本 |
| 重大新闻事件 | 新闻搜索API、维基百科事件 | 搜索+筛选 | 还原时代背景 |

### 10.3 查询触发时机

用户触发推演时发起查询，不预先爬取：

```
场景：用户问"如果当初去了深圳会怎样"
                │
                ▼
系统提取时间点（用户提到的时间）
                │
                ▼
系统提取地点（深圳）
                │
                ▼
并行查询：
· 2024年深圳房价走势
· 2024年深圳空气质量
· 2024年深圳相关政策变化
· 2024年深圳重大新闻事件
                │
                ▼
结果整合 → 呈现给用户
```

### 10.4 数据呈现方式

不是堆砌数据，而是融入对话：

```
用户问："如果当初去了那家公司会怎样"

系统查询后发现该公司在用户拒绝后6个月裁员30%

呈现方式：
"我查了一下，
 你当时放弃的那家公司，在2024年3月做了一次架构调整，
 裁掉了差不多30%的人——主要是你差点去的那个部门。

 这不是说你当时的选择是对的，
 只是说，那条路可能没有你想象的那么稳。"
```

### 10.5 数据缓存与更新

| 数据类型 | 缓存策略 | 更新频率 |
|---------|---------|---------|
| 天气历史 | 永久缓存（历史数据不变） | 无需更新 |
| 空气质量历史 | 永久缓存 | 无需更新 |
| 公司动态 | 缓存30天 | 用户再次查询时检查更新 |
| 房价走势 | 缓存7天 | 每周更新 |
| 新闻事件 | 缓存90天 | 用户再次查询时检查更新 |

## 十一、与情景推演引擎接口

### 11.1 接口定位

- 长期对话系统 = "理解你"
- 情景推演引擎 = "推演另一条路"

接口职责：
- 为推演引擎提供用户画像数据
- 为推演引擎提供相关记忆片段
- 将推演结果存回记忆库
- 根据推演反馈更新画像

### 11.2 数据流向

```
┌─────────────┐                      ┌─────────────────┐
│ 长期对话系统 │                      │ 情景推演引擎     │
│             │  推演请求 + 画像数据  │                 │
│ ┌─────────┐ │ ──────────────────────▶│ ┌─────────────┐ │
│ │ 用户画像 │ │                      │ │ 推演上下文   │ │
│ └─────────┘ │  相关记忆片段          │ └─────────────┘ │
│ ┌─────────┐ │ ──────────────────────▶│ ┌─────────────┐ │
│ │ 记忆库   │ │                      │ │ 场景构建     │ │
│ └─────────┘ │                      │ └─────────────┘ │
│             │◀──────────────────────│                 │
│             │  推演结果存回记忆库     │                 │
│             │◀──────────────────────│                 │
│             │  用户反馈更新画像       │                 │
└─────────────┘                      └─────────────────┘
```

### 11.3 推演请求结构

```json
{
  "request_id": "sim_20260421_xxxx",
  "type": "parallel_life | decision_analysis",
  "trigger": {
    "user_statement": "如果当初没有辞职会怎样",
    "time_point": "2024-03",
    "choice_made": "辞职",
    "choice_not_made": "留下"
  },
  "required_profile_dimensions": [
    "价值观.工作观",
    "决策原则",
    "风险承受底线",
    "能力与资源边界",
    "关系图谱.工作相关"
  ],
  "required_memory_types": [
    "关键决策片段",
    "时间线事件.2024年",
    "遗憾线索.职业相关"
  ],
  "external_data_needed": [
    "公司动态.前公司",
    "行业变化.2024年"
  ]
}
```

### 11.4 推演上下文构建

长期对话系统根据推演请求，构建推演上下文：

```
【用户画像摘要】
· 工作观：追求成长而非稳定，但害怕不确定性
· 决策风格：犹豫后行动，行动后常怀疑
· 风险承受：中等偏低，有房贷压力
· 关键关系：配偶支持但担心收入波动

【当年情境还原】（从记忆库提取）
· 时间：2024年3月
· 当时状态：工作3年，感觉瓶颈，想跳槽
· 决策因素：有创业offer但风险大，最终选择辞职休息
· 关键记忆片段：（原文引用）
  "那时候真的累了，每天上班像上坟……"
  "创业那个offer其实挺诱人的，但房贷不敢赌"

【外部真实数据】
· 前公司：2024年9月裁员20%
· 创业公司：2024年底B轮融资成功
· 行业：2024年大环境偏冷，但该赛道逆势增长
```

### 11.5 推演结果存储

推演完成后，结果存回记忆库：

```json
{
  "type": "simulation_result",
  "request_id": "sim_20260421_xxxx",
  "created_at": "2026-04-21T15:30:00",
  "scenario": "如果当初没有辞职",
  "user_reality": "辞职休息后自由职业",
  "simulation_summary": "如果留下，可能经历裁员风波……",
  "key_insights": [
    "当时的选择并非完全错误",
    "未选择的路同样存在风险"
  ],
  "user_feedback": null,
  "emotional_impact": null
}
```

### 11.6 用户反馈与画像更新

推演呈现后，系统自然询问用户反馈，根据反馈更新画像：

| 反馈类型 | 画像更新动作 |
|---------|-------------|
| "很符合我的想法" | 强化相关画像维度 |
| "不太对" | 记录偏差，可能需要画像校准 |
| "没想到这一点" | 新增认知，更新自我叙事模式 |
| "看完后觉得……" | 记录情绪反应，更新情感模式 |
| 用户补充新信息 | 更新记忆库，修正推演依据 |

### 11.7 两种推演模式的接口差异

| 维度 | 平行人生推演 | 决策辅助分析 |
|-----|-------------|-------------|
| 触发 | 用户主动询问"如果当初……" | 用户面临重大决策，请求帮助分析 |
| 数据需求 | 历史记忆 + 外部真实数据 | 当前画像 + 外部真实数据 + 行业/市场信息 |
| 推演方向 | 过去的另一种可能 | 未来的多种可能性 |
| 呈现重点 | 那条路上的风景和泥泞 | 客观的利弊分析、概率推演、参照案例、行动建议 |
| LLM能力调用 | 历史数据查询与整合 | 过往决策案例检索、行业趋势分析、概率模型计算 |
| 更新画像 | 遗憾线索、自我叙事模式 | 风险承受底线、决策原则、能力边界认知 |

### 11.8 决策辅助分析模块详解

#### 11.8.1 设计原则

**核心定位：客观推演助手，不是对抗式挑战者**

决策辅助分析的目标是帮助用户做出更理性的决策，而不是"施压"或"反驳"。系统站在用户的视角，基于用户画像、客观事实和LLM的分析能力，提供全面的信息支持和建议。

**三大支柱：**

```
│ 分析输出：                                                       │
│ "你的配偶之前表示过担心你的职业稳定性                            │
│  建议在做决定前与她深入沟通，让她参与分析过程                     │
│  这不是关于说服，而是关于共同面对不确定性"                        │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 11.8.6 分析后的画像更新

```
决策辅助分析完成后，系统自动提取洞察更新画像：

┌─────────────────────────────────────────────────────────────────┐
│ 画像维度            │ 更新内容                                   │
├─────────────────────┼───────────────────────────────────────────┤
│ 决策原则            │ 用户最终选择的依据 → 揭示真实决策权重       │
│ 风险承受底线        │ 用户对风险的反应 → 校准风险承受基线         │
│ 自我叙事模式        │ 用户如何讲述这个决定 → 更新叙事风格         │
│ 关系图谱            │ 决策中参考了谁的意见 → 更新关系影响权重     │
│ 能力边界认知        │ 用户识别的能力缺口 → 更新能力边界           │
└─────────────────────────────────────────────────────────────────┘

示例更新：
· 用户最终选择不接受offer
· 更新原因："家庭因素优先于职业成长"
· 画像更新：价值观.家庭权重上调
· 画像更新：决策风格.风险厌恶度上调
```

## 十二、多端同步机制

### 12.1 同步范围

**端：** App（iOS/Android）+ Web

**同步内容：**
- 用户画像（完整）
- 记忆库（原始对话、关键片段、时间线）
- 隐藏层记忆
- 用户设置（对话风格偏好、授权配置等）
- 推演历史

**不同步内容：**
- 本地缓存（如临时会话状态）
- 设备特定设置（如通知偏好）

### 12.2 同步架构

```
           ┌─────────────────────┐
           │   云端加密存储       │
           │  (只存密文，不解密)   │
           └──────────┬──────────┘
                      │
          ┌───────────┴───────────┐
          │                       │
          ▼                       ▼
   ┌─────────────┐         ┌─────────────┐
   │ App (本地)   │         │ Web (本地)  │
   │             │         │             │
   │ · 完整记忆库 │         │ · 完整记忆库 │
   │ · 完整画像   │         │ · 完整画像   │
   │ · Keychain   │         │ · IndexedDB │
   │   存密钥     │         │   存密钥     │
   └─────────────┘         └─────────────┘
```

**同步原则：**
- 本地优先：所有数据先写本地，再同步云端
- 加密传输：传输前用用户密钥加密，云端只存密文
- 冲突解决：以时间戳+操作类型决定最终状态

### 12.3 同步触发时机

| 优先级 | 触发时机 | 数据类型 |
|-------|---------|---------|
| 高 | 实时同步 | 用户画像更新、关键片段生成、时间线事件添加、用户纠正 |
| 中 | 定时同步 | 原始对话记录（每5分钟或会话结束时）、周期校准结果 |
| 低 | 延迟同步 | 压缩后的历史对话摘要、使用统计数据 |
| 手动 | 用户触发 | 用户主动触发"立即同步"、切换设备时自动检测 |

### 12.4 冲突解决策略

| 冲突类型 | 解决策略 |
|---------|---------|
| 画像同一维度冲突 | 以最新时间戳为准，但记录冲突历史，用户可在设置中查看 |
| 记忆片段冲突 | 合并而非覆盖，两边的内容都保留，标记来源 |
| 删除冲突 | 删除优先，一端删除后同步后另一端也删除，但进入隐藏层 |
| Correction冲突 | 以最新为准，但记录所有Correction |

**冲突通知：** 当发生冲突时，系统在下次对话开始时自然告知用户。

### 12.5 同步状态管理

| 状态 | 图标 | 用户感知 |
|-----|------|---------|
| 已同步 | ✓ | 无需提示 |
| 同步中 | ↻ | 静默显示，不打断对话 |
| 离线 | ○ | 提示"离线模式，稍后同步" |
| 同步失败 | ! | 提示"同步遇到问题，点击重试" |
| 冲突待处理 | ⚠ | 提示"有冲突需要确认" |

**状态显示位置：**
- App：设置页面顶部状态栏
- Web：页面右下角状态指示器

### 12.6 离线模式

**离线时：**
- 所有数据写入本地，标记"待同步"
- 对话、画像更新、记忆存储均正常工作
- 功能不受影响，只是无法跨设备

**恢复在线时：**
- 自动检测待同步数据
- 按优先级依次同步
- 处理冲突
- 同步完成后静默通知用户

### 12.7 首次登录/新设备登录

```
用户输入账号密码
        │
        ▼
验证通过，云端返回加密数据
        │
        ▼
用户输入主密码（或生物识别）解密密钥
        │
        ▼
本地解密数据，写入本地存储
        │
        ▼
同步完成，可以开始使用
```

### 12.8 数据同步版本控制

每个数据对象携带版本信息：

```json
{
  "id": "profile_values",
  "version": 12,
  "updated_at": "2026-04-21T15:30:00Z",
  "updated_device": "app_ios",
  "checksum": "sha256:abc123..."
}
```

版本控制规则：
- 每次更新版本号+1
- 同步时比较版本号，高版本覆盖低版本
- 相同版本号时比较时间戳
- 校验checksum确保数据完整性

## 十三、设计决策记录

| 决策点 | 选择 | 理由 |
|-------|------|------|
| 数据架构 | 双核驱动（记忆库+画像层） | 兼顾完整性和可用性，纠偏有据可依 |
| 存储架构 | 本地计算+云端加密存储 | 隐私最大化，服务端无法读取 |
| 关系图谱粒度 | 深度记录（关系人、权重、典型互动模式、关键事件、情绪变化上下文） | 支撑深度推演，但不过度复杂 |
| 画像建立方式 | 默认对话+支持数据导入 | 混合策略，灵活快速 |
| 初始对话方式 | 单线深入式，分多次完成 | 体验即产品，不设模块 |
| 对话风格 | 默认哲学向导，自动调整 | 根据用户画像适配 |
| 用户删除记忆 | 进入隐藏层，不是物理删除 | "想删除什么"更能暴露真实心理 |
| 删除原因询问 | 对话式，柔性询问 | 保持拟人交互体验 |
| 周期校准触发 | 智能触发为主，定期提醒为辅 | 自然不过度打扰 |
| 记忆保留策略 | 分层保留+用户可配置 | 关键片段永久，原始对话2年后压缩 |
| 幽灵日历数据来源 | 对话自动提取+用户主动记录+日历接入 | 多渠道覆盖 |
| 外部数据接入策略 | 前期仅公开信息 | 避免隐私风险，后续可扩展 |
| 公司/岗位动态数据 | 仅公开新闻报道、公司公告 | 避免职场社交数据敏感性 |
| 多端范围 | App + Web（暂不含微信机器人） | 前期聚焦核心体验 |
