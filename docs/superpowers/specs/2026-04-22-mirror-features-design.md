## 二、用户画像（冷启动入口）

### 2.1 设计目标

用户冷启动的"哇"时刻：让探索型用户快速感知产品价值，建立对产品的兴趣和信任。

**核心洞察：**
- 当下用户对性格测试、MBTI、各种测评有天然兴趣
- "了解自己"是普世的、低门槛的需求
- 画像是产品与用户建立连接的第一触点

### 2.2 画像生成路径

```
┌─────────────────────────────────────────────────────────────────┐
│                      画像生成路径选择                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  【路径一：数据导入】                                            │
│  ─────────────────────────────────────────────────────────────  │
│  导入AI聊天记录 → 快速分析 → 生成画像                           │
│  优点：快速、信息丰富                                           │
│  前提：用户有可导入的数据                                       │
│                                                                 │
│  【路径二：快问快答】                                            │
│  ─────────────────────────────────────────────────────────────  │
│  5-10个问题 → 快速出画像                                        │
│  优点：门槛低、互动感强                                         │
│  问题：信息量有限，准确性依赖问题设计                           │
│                                                                 │
│  【路径三：混合路径】（推荐）                                     │
│  ─────────────────────────────────────────────────────────────  │
│  有数据 → 导入 + 少量澄清问题 → 画像                            │
│  无数据 → 快问快答 → 初步画像 → 后续对话逐步完善                │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 三、关键事项记录

### 3.1 功能定位

**核心思路转变：**

从"被动挖掘遗迹"转变为"主动记录关键事项"。

| | 被动挖掘（旧思路） | 主动记录（新思路） |
|---|---|---|
| 用户动作 | 导入数据，系统解析 | 主动标记或对话中记录 |
| 信息质量 | 噪音多，需要筛选 | 用户筛选过，质量高 |
| 隐私问题 | 复杂，涉及第三方信息 | 用户主动，边界清晰 |
| 技术难度 | 高，OCR、解析、理解 | 低，理解用户输入即可 |
| 用户心智 | "系统知道我什么" | "我告诉系统什么" |

**什么是"关键事项"？**

用户人生中的关键时刻：

- 做了一个重要决定（换工作、搬家、分手、结婚）
- 经历了一个重要事件（入职、离职、旅行、生病）
- 有一个重要的想法转折（想通了某事、改变了看法）
- 差点发生但没有发生的事（差点去的公司、差点买的车）

**为什么用户愿意记录？**

价值飞轮：

```
记录关键事项 → 推演更准确 → 用户感知到价值 → 想要更准的推演 → 愿意记录更多
```

### 3.2 记录什么？

```
┌─────────────────────────────────────────────────────────────────┐
│                      关键事项分类                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  【决定类】                                                      │
│  ─────────────────────────────────────────────────────────────  │
│  重大选择、改变方向的决定                                        │
│  示例：决定辞职、决定转行、决定搬到另一个城市                    │
│                                                                 │
│  记录内容：                                                      │
│  · 决定是什么                                                    │
│  · 为什么做这个决定                                              │
│  · 当时考虑的其他选项                                            │
│  · 期望的结果是什么                                              │
│                                                                 │
│  【事件类】                                                      │
│  ─────────────────────────────────────────────────────────────  │
│  重要的里程碑事件                                                │
│  示例：入职新公司、完成一个项目、结束一段关系                    │
│                                                                 │
│  记录内容：                                                      │
│  · 事件是什么                                                    │
│  · 发生了什么                                                    │
│  · 对你意味着什么                                                │
│                                                                 │
│  【转折类】                                                      │
│  ─────────────────────────────────────────────────────────────  │
│  想法、心态、价值观的转变                                        │
│  示例：想通了某件事、改变了某个看法                              │
│                                                                 │
│  记录内容：                                                      │
│  · 之前怎么想                                                    │
│  · 之后怎么想                                                    │
│  · 为什么转变                                                    │
│                                                                 │
│  【差点发生类】（幽灵事件）                                      │
│  ─────────────────────────────────────────────────────────────  │
│  差点发生但没有发生的事                                          │
│  示例：差点去的公司、差点买的车、差点在一起的人                  │
│                                                                 │
│  记录内容：                                                      │
│  · 差点发生什么                                                  │
│  · 为什么没发生                                                  │
│  · 现在还怎么想                                                  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 3.3 如何让记录足够轻便？

**设计原则：不增加用户的额外负担**

```
┌─────────────────────────────────────────────────────────────────┐
│                      记录方式设计                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  【方式一：对话中自动识别】（主要方式）                          │
│  ─────────────────────────────────────────────────────────────  │
│  用户正常对话 → 系统识别关键事项 → 自动提取记录                  │
│                                                                 │
│  示例：                                                          │
│  用户："我最近在考虑要不要换工作，现在这家公司越来越累了"        │
│  系统：（识别为"决定类"待定）记录：考虑换工作                    │
│                                                                 │
│  后续用户："决定了，我准备下个月提离职"                          │
│  系统：（更新记录）决定：离职                                    │
│        （询问）"要记录一下这个决定吗？你想离开的原因是什么？"     │
│                                                                 │
│  优点：零额外操作，自然融入对话                                  │
│  注意：用户可以说"这个不用记"                                    │
│                                                                 │
│  ─────────────────────────────────────────────────────────────  │
│                                                                 │
│  【方式二：一键标记】                                            │
│  ─────────────────────────────────────────────────────────────  │
│  对话中某句话 → 用户点击"标记"或说"记下来"                       │
│                                                                 │
│  示例：                                                          │
│  用户："这件事对我来说挺重要的，记一下"                          │
│  或：用户点击某句话旁边的标记按钮                                │
│  系统：记录了，我把它加入关键事项                                │
│                                                                 │
│  优点：用户有明确控制感                                          │
│                                                                 │
│  ─────────────────────────────────────────────────────────────  │
│                                                                 │
│  【方式三：主动记录】                                            │
│  ─────────────────────────────────────────────────────────────  │
│  用户主动发起记录某个事项                                        │
│                                                                 │
│  示例：                                                          │
│  用户："记录一件事：我决定搬到杭州了"                            │
│  系统：好的，我记下了。                                          │
│        "是什么让你决定搬过去的？"                                │
│        （通过对话补充细节）                                      │
│                                                                 │
│  优点：适合想回顾时主动记录                                       │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 3.4 重要性判断与深入沟通

**核心设计：不是所有记录都同等对待**

系统在识别到关键事项时，会判断其重要性。对于重要记录，系统会试探性地深入沟通，获取更多信息后再"存档"。

```
┌─────────────────────────────────────────────────────────────────┐
│                      重要性判断维度                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  【类型权重】                                                    │
│  ─────────────────────────────────────────────────────────────  │
│  决定类：高权重（影响人生轨迹）                                  │
│  转折类：高权重（价值观变化）                                    │
│  事件类：中权重（取决于事件影响程度）                            │
│  差点发生类：中低权重（取决于遗憾程度）                          │
│                                                                 │
│  【内容判断】                                                    │
│  ─────────────────────────────────────────────────────────────  │
│  · 涉及用户画像中的核心价值观                                    │
│  · 用户情绪表达强烈（"终于""再也不""一直"等强调词）             │
│  · 涉及用户画像中的"遗憾线索"                                   │
│  · 涉及重要关系人（关系图谱中高权重的人）                        │
│  · 时间跨度大或影响深远的决定                                    │
│                                                                 │
│  【行为信号】                                                    │
│  ─────────────────────────────────────────────────────────────  │
│  · 用户主动说"记下来"                                           │
│  · 用户反复提及同一件事                                          │
│  · 用户在对话中花大量时间讨论                                    │
│                                                                 │
│  判断结果：                                                      │
│  · 高重要性 → 触发深入沟通                                      │
│  · 中重要性 → 简单确认后记录                                    │
│  · 低重要性 → 静默记录，不打断对话                              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

```

│  用户："我想了很长时间，决定辞职了。"                            │
│                                                                 │
│  系统：（判断：决定类 + 强调词"很长时间" + 情绪信号 → 高重要性） │
│                                                                 │
│  系统："这是一个重要的决定。                                     │
│        去哪边？"                                                 │
│                                                                 │
│  用户："还没想好，先休息一段时间。"                              │
│                                                                 │
│  系统："休息多久？后面的打算呢？"                                │
│                                                                 │
│  用户："可能两三个月吧。想趁这段时间好好想想接下来要做什么。"    │
│                                                                 │
│  系统："那这段时间的经济压力呢？"                                │
│                                                                 │
│  用户："还好，存款够撑半年。"                                    │
│                                                                 │
│  系统："行。                                                    │
│        辞职，裸辞，休息两三个月，存款够半年。                    │
│        我记下来了。"                                            │
│                                                                 │
│  【存档动效触发】                                                │
│                                                                 │

### 3.8 节点的不可变性与追溯机制

**核心原则：客观事实一旦记录，不可随意修改或推翻**

```

│  · ✅ 可以添加关联的其他信息                                    │
│  · ❌ 不可直接修改核心事实                                      │
│  · ❌ 不可删除后"假装没发生"                                    │

│  · 时间线是用户人生的客观记录                                    │

│                                                                 │

│                                                                 │
│  ─────────────────────────────────────────────────────────────  │
│                                                                 │
│  【如果用户不想多说】                                            │
│  ─────────────────────────────────────────────────────────────  │
│  用户："不用问了，直接删吧"                                     │
│  系统："好。"                                                   │
│       （节点移入隐藏层，标记为"用户回避深入讨论"）               │

**隐藏层的作用（与第一阶段设计呼应）：**

```

│                      隐藏层设计意图                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │

│  ─────────────────────────────────────────────────────────────  │

│  · 删除原因本身就是心理洞察                                      │

```

## 四、遗憾管理

### 4.1 功能定位

**核心目标：守护人生优先级，减少未来的遗憾**

遗憾管理不是消除遗憾，而是：

- 帮助用户识别真正重要的遗憾
- 区分"可弥补"和"不可弥补"的遗憾
- 对"可弥补"的遗憾，推动行动
- 对"不可弥补"的遗憾，帮助释怀

### 4.6 可弥补遗憾的处理

**核心原则：帮助用户看清自己，而不是强行改变**

对于"可弥补"的遗憾，系统不是简单地推动行动，而是基于用户画像和历史进行综合推演。

```
┌─────────────────────────────────────────────────────────────────┐
│                    推动力度的判断依据                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  【输入一：遗憾程度】                                            │
│  ─────────────────────────────────────────────────────────────  │
│  高遗憾程度 → 更需要关注                                        │
│  中/低遗憾程度 → 适度关注                                       │
│                                                                 │
│  【输入二：历史参照】                                            │
│  ─────────────────────────────────────────────────────────────  │
│  · 类似遗憾但已弥补的历史 → 提取成功路径作为参考                 │
│  · 类似遗憾但未弥补的历史 → 提取失败教训作为提醒                 │
│                                                                 │
│  示例：                                                          │
│  用户遗憾：没有学设计（高遗憾程度）                              │
│  历史参照：用户之前"想健身"最终做到了（成功路径）                 │
│  参考路径：先定小目标、找同伴、每周固定时间                      │
│                                                                 │
│  历史参照：用户之前"想学吉他"放弃了（失败教训）                   │
│  失败原因：没时间、没有反馈                                      │
│  避免重蹈：这次要注意时间安排和反馈机制                          │
│                                                                 │
│  【输入三：用户画像】                                            │
│  ─────────────────────────────────────────────────────────────  │
│  · 行动偏好：用户是计划型还是随性型                              │
│  · 时间资源：用户当前的时间压力                                  │
│  · 风险承受：用户对失败的容忍度                                  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**疲劳度机制：防止重复打扰**

```
┌─────────────────────────────────────────────────────────────────┐
│                      疲劳度控制机制                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  【触发记录】                                                    │
│  ─────────────────────────────────────────────────────────────  │
│  每次主动提及某个遗憾，记录一次触发                              │
│                                                                 │
│  【疲劳度递增】                                                  │
│  ─────────────────────────────────────────────────────────────  │
│  · 第1次提及：正常跟进                                          │
│  · 第2次提及：间隔至少7天                                       │
│  · 第3次提及：间隔至少14天                                      │
│  · 第4次提及：间隔至少30天                                      │
│  · 用户明确表示"别问了"：暂停提及该遗憾                         │
│                                                                 │
│  【疲劳度归零】                                                  │
│  ─────────────────────────────────────────────────────────────  │
│  · 用户主动提起该遗憾                                           │
│  · 用户对该遗憾有了新进展                                       │
│  · 用户表示想继续跟进                                           │
│                                                                 │
│  【触发时机 - 分类型处理】                                      │
│  ─────────────────────────────────────────────────────────────  │
│                                                                 │
│  【不可弥补型遗憾】                                              │
│  触发条件：对话中出现与该遗憾相关的场景                          │
│                                                                 │
│  示例：                                                          │
│  遗憾：没有好好陪奶奶（已去世）                                  │
│  相关场景：用户提到奶奶、家庭、亲情、遗憾、时光等                │
│                                                                 │
│  用户："今天妈妈打电话说外公身体不太好"                          │
│  系统：（检测到与"亲情"相关场景）                               │
│        "听到这些，会不会让你想起之前的事？"                      │
│        （若疲劳度允许，自然关联到遗憾话题）                      │
│                                                                 │
│  频次控制：                                                      │
│  · 首次提及后，间隔至少14天才能再次主动提及                      │
│  · 若用户表示"不想聊这个"，该遗憾进入静默状态                   │
│  · 静默状态持续30天后，才可能在相关场景下再次触发                │
│                                                                 │
│  反馈衰减/强化机制：                                             │
│  ─────────────────────────────────────────────────────────────  │
│  · 用户主动深入聊 → 下次可更早触发                              │
│  · 用户表示"不想聊" → 进入静默，衰减触发频率                    │
│  · 用户表示"已经释怀了" → 标记为已释怀，不再主动触发            │
│                                                                 │
│  ─────────────────────────────────────────────────────────────  │
│                                                                 │
│  【可弥补型遗憾】                                                │
│  触发条件：时间跨度 + 可弥补周期 + 当前状态                      │
│                                                                 │
│  时间跨度判断：                                                  │
│  · 短期遗憾（如"想学设计"）：可弥补窗口较短                      │
│  · 长期遗憾（如"想陪家人旅行"）：可弥补窗口较长                  │
│                                                                 │
│  智能触发逻辑：                                                  │
│  ─────────────────────────────────────────────────────────────  │
│  1. 分析可弥补周期                                              │
│     · 学设计：需要数月到数年，窗口长                             │
│     · 报名课程：需要数周，窗口短                                 │
│     · 陪家人旅行：需要假期+金钱，窗口中等                        │
│                                                                 │
│  2. 计算触发紧迫度                                               │
│     紧迫度 = 遗憾程度 × 可弥补窗口剩余比例                        │
│     · 窗口快关闭 → 紧迫度高                                     │
│     · 窗口充足 → 紧迫度低                                       │
│                                                                 │
│  3. 选择触发时机                                                 │
│     · 紧迫度高 → 更主动地提及                                   │
│     · 紧迫度低 → 等待自然关联场景                               │
│                                                                 │
│  示例：                                                          │
│  ─────────────────────────────────────────────────────────────  │
│  遗憾：想学设计（提出于3个月前，高遗憾程度）                      │
│  可弥补窗口：假设5年内都可能学                                   │
│  当前紧迫度：中等（窗口充足，但时间已过3个月）                   │
│  触发策略：等待自然关联场景，不过度主动                          │
│                                                                 │
│  用户："最近想学点新东西"                                       │
│  系统：（自然关联场景）                                         │
│        "你之前提过想学设计，最近还有这个想法吗？"                │
│                                                                 │
│  ─────────────────────────────────────────────────────────────  │
│  遗憾：报名某个即将截止的课程（提出于2周前，截止日期临近）        │
│  可弥补窗口：剩余2周                                             │
│  当前紧迫度：高                                                  │
│  触发策略：主动提醒                                              │
│                                                                 │
│  系统："上次你说的那个课程，报名好像快截止了。                   │
│         你还想报吗？"                                            │
│                                                                 │
│  【通用规则】                                                    │
│  ─────────────────────────────────────────────────────────────  │
│  · 不在用户打开APP时主动弹窗                                    │
│  · 只在对话中自然关联或智能判断适当时机提及                      │
│  · 疲劳度机制始终生效，任何类型的遗憾都要遵守                     │
│  · 用户明确拒绝后，进入静默状态                                  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 6.1 与长期对话系统的接口

```
┌─────────────────────────────────────────────────────────────────┐
│                  第三阶段 ←→ 第一阶段 接口                       │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  【记忆库接口】                                                  │
│  ─────────────────────────────────────────────────────────────  │
│  写入：                                                         │
│  · 数字遗迹还原结果 → 记忆库.时间线层                           │
│  · 幽灵事件 → 记忆库.时间线层.未发生事件时间线                  │
│  · 遗憾记录 → 记忆库.关键片段层                                 │
│                                                                 │
│  读取：                                                         │
│  · 用户画像 → 用于遗迹解读、遗憾分析                            │
│  · 历史对话 → 用于遗憾识别、情境还原                            │
│  · 关联记忆 → 用于推演素材准备                                  │
│                                                                 │
│  ─────────────────────────────────────────────────────────────  │
│                                                                 │
│  【用户画像接口】                                                │
│  ─────────────────────────────────────────────────────────────  │
│  读取：                                                         │
│  · 价值观 → 用于遗憾分类、行动建议                              │
│  · 遗憾线索库 → 直接对接遗憾管理                                │
│  · 人生阶段状态 → 用于遗憾优先级判断                            │
│  · 关系图谱 → 用于涉及关系的遗憾处理                            │
│                                                                 │
│  写入：                                                         │
│  · 遗憾模式发现 → 画像.行为偏好                                 │
│  · 遗憾处理结果 → 画像.遗憾线索库                               │
│  · 行动完成情况 → 画像.能力边界                                 │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```
